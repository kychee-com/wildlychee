## ADDED Requirements

### Requirement: Hero section reserves layout space

The `.section-hero` SHALL have `min-height: 60vh` when a background image is configured. This prevents layout shift when the hero image loads or when hero content is injected dynamically.

#### Scenario: Hero with background image
- **WHEN** a hero section has a `bg_image` configured
- **THEN** the section occupies at least 60vh from first paint, before the image loads

#### Scenario: Hero without background image
- **WHEN** a hero section has no `bg_image`
- **THEN** the section height is determined by its content and padding (no forced min-height)

### Requirement: Sections container reserves layout space

The `#sections` container SHALL have `min-height: 50vh` to prevent layout shift when sections are fetched and injected. The min-height SHALL be removed after sections finish rendering.

#### Scenario: Sections loading
- **WHEN** the homepage is loading sections from the API
- **THEN** the `#sections` container occupies at least 50vh, preventing content below from shifting upward

#### Scenario: Sections rendered
- **WHEN** all sections have been rendered into the DOM
- **THEN** the `min-height` constraint is removed and the container sizes to its content

### Requirement: Image elements have explicit dimensions

All `<img>` elements generated by JavaScript (member avatars, event images, announcement images) SHALL include `width` and `height` attributes matching their rendered size. Image containers SHALL use CSS `aspect-ratio` to reserve space before images load.

#### Scenario: Avatar images have dimensions
- **WHEN** a member avatar `<img>` is created in the directory or forum
- **THEN** it has `width` and `height` attributes (e.g., `width="48" height="48"`)

#### Scenario: Event card images have dimensions
- **WHEN** an event card image `<img>` is created
- **THEN** it has `width` and `height` attributes matching the card image dimensions

### Requirement: Nav user area builds atomically

The `#nav-user` element SHALL be populated in a single DOM write (one `innerHTML` assignment) containing the auth UI, theme toggle, and language switcher. This prevents multiple reflows from staged construction.

#### Scenario: Logged-in user nav renders
- **WHEN** a logged-in user's nav area is built
- **THEN** the avatar, sign-out button, theme toggle, and language switcher are all inserted in one DOM operation

#### Scenario: Anonymous user nav renders
- **WHEN** an anonymous user's nav area is built
- **THEN** the sign-in button and theme toggle are inserted in one DOM operation

### Requirement: Parallel homepage data fetches

The homepage SHALL fetch sections, announcements, and activity feed data in parallel using `Promise.all()` instead of sequential awaits.

#### Scenario: Homepage loads data
- **WHEN** the homepage initializes after config is loaded
- **THEN** sections, announcements, and activity_log API calls are dispatched simultaneously
- **THEN** each section renders as its data arrives without waiting for others
