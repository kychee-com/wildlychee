## Context

Lighthouse measures CLS and LCP on cold loads (no cache, simulated mobile). Our current scores:

| Site | Score | CLS | LCP |
|------|-------|-----|-----|
| Eagles | 75 | 0.775 | 2.0s |
| Silver Pines | 50 | 0.95 | 10.8s |
| Barrio Unido | 52 | 0.576 | 11.6s |

**CLS root causes** (each adds ~0.1-0.3 to the score):
1. Sections start `opacity: 0; transform: translateY(20px)` — visually hidden but also collapses layout height. When they become visible, everything below shifts.
2. Homepage sections are fetched from the API and injected into `#sections` — the container has no reserved height, so content pops in.
3. Nav user area builds in 3 stages (auth UI → theme toggle → language switcher), each reflow.
4. Hero section has no `min-height` — shifts as background image loads.

**LCP root causes** (10+ seconds on 2 of 3 sites):
1. Hero background image loads through a 3-hop chain: HTML → config.js init() → sections API fetch → CSS `background-image` set → image download. No preloading.
2. Homepage API calls are sequential (`await renderSections()` then `loadActivityFeed()` then `await renderAnnouncements()`).

## Goals / Non-Goals

**Goals:**
- CLS < 0.1 on all 3 sites (from 0.5-0.95)
- LCP < 2.5s on all 3 sites (from 2-11.6s)
- Lighthouse performance score 90+ on all 3 sites
- No visual regression — pages should look identical after load completes

**Non-Goals:**
- Image format optimization (WebP/AVIF) — requires Run402 image transformation support
- Server-side rendering or static generation
- Changing the scroll animation visual effect (still want fade-in, just without layout shift)

## Decisions

### 1. Use `visibility: hidden` + fixed dimensions instead of `opacity: 0` for section animations

**Choice**: Sections start `visibility: hidden; opacity: 0` but keep their full layout dimensions (padding, min-height). The `section-visible` class sets `visibility: visible; opacity: 1` with a CSS transition.

**Why**: `visibility: hidden` reserves layout space (unlike `display: none`). Combined with explicit padding/min-height, sections occupy their final space from first paint. The opacity transition still provides the visual fade-in effect.

**Trade-off**: The `translateY(20px)` slide-up effect must be removed because it shifts layout. Pure opacity fade is simpler and CLS-safe.

### 2. Preload hero image from cached config

**Choice**: In `config.js`, after reading `site_config` from cache, immediately inject a `<link rel="preload" as="image" href="...">` into `<head>` for the hero background image URL. This starts the download in parallel with API calls instead of waiting for the sections fetch.

**Why**: The hero image URL is in `site_config.nav` or `sections` data. On warm loads, it's in the localStorage cache. Preloading it 500-1000ms earlier is the difference between 2s LCP and 10s LCP.

**Fallback**: On cold load (no cache), the preload can't happen until after the `site_config` fetch. But the cache-first change from the previous PR means most real users will have warm caches.

### 3. Parallelize homepage data fetches

**Choice**: Replace sequential `await` calls with `Promise.all()`:
```js
// Before: sequential
await renderSections();
loadActivityFeed(15);
await renderAnnouncements();

// After: parallel
await Promise.all([renderSections(), renderAnnouncements(), loadActivityFeed(15)]);
```

**Why**: These 3 calls are independent (sections, announcements, activity_log). Running them in parallel cuts the total wait from sum-of-latencies to max-of-latencies, saving 200-500ms.

### 4. Set min-height on hero and sections container

**Choice**: CSS `min-height` on `.section-hero` (60vh) and `#sections` container (based on typical content). The skeleton in `#sections` already provides some height but collapses too early.

**Why**: Even before JS runs, the page has a stable layout. Content loads into reserved space instead of expanding from zero.

### 5. Set explicit dimensions on avatar and card images

**Choice**: Add `width` and `height` attributes to all `<img>` elements generated by JS (member avatars, event images). Use CSS `aspect-ratio` on image containers where dimensions aren't known at HTML-write time.

**Why**: Browsers use width/height to calculate aspect ratio before image loads, preventing reflow.

## Risks / Trade-offs

**[Removed slide-up animation]** → The `translateY(20px)` slide effect causes CLS by definition. Removing it means sections simply fade in without upward motion. This is a minor visual regression but eliminates ~0.2 CLS per section.

**[Hero min-height on short content]** → If a site has a very short hero, `min-height: 60vh` could look oversized. Mitigation: use `min-height` only when a background image is present, otherwise let content determine height.

**[Preload on cold load is limited]** → Without cached config, we can't preload the hero image early. The first visit will still have slower LCP. Mitigation: acceptable since the cache-first work means repeat visits are the common case.
