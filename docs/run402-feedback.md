# Run402 Platform Feedback — Kychon

Per saas-factory F27 ("Live Platform-Gaps Feedback Loop"), this file tracks run402 platform capabilities Kychon needs but doesn't yet have. Each numbered paragraph covers the gap, current workaround, suggested fix, and priority. Items marked **✅ FIXED** have been closed upstream; they stay in the record for historical context — closed-loop evidence that product demand drove a platform change.

See also: `docs/spec.md` § "Run402 Platform Gaps" for additional context on items 4-11 (this file is the canonical F27 record; spec.md is kept for in-context reference during product design).

---

1. **Auto-archive without warning — BLOCKER.** Run402 auto-archives projects 7 days after tier lease expiry (`packages/gateway/src/services/leases.ts`, hourly cron). The archive cascade hard-drops the tenant schema (`DROP SCHEMA ... CASCADE` — unrecoverable), releases subdomains, leaves `internal.domains` rows partially orphaned, and marks `status='archived'`. There is no warning email before archive, no soft-delete window, and no restore endpoint anywhere in the platform. Kychon hit this on 2026-04-14: all 4 projects (eagles, silver-pines, barrio, marketing) were silently archived after the prototype tier's 7-day lease expired on 2026-04-06 and the checker's 7-day grace elapsed. Recovery required re-provisioning 4 new projects with new `project_id`s, updating `.env`, and using admin SQL to surgically rebind orphaned custom-domain rows. **Workaround:** `run402 projects pin <id>` every production project immediately after `projects provision`; monitor `tier status` manually. **Suggested fix:** (a) warning email 7d + 24h before archive, (b) 30-day soft-delete window with `POST /projects/v1/admin/:id/restore`, (c) refuse-to-archive for projects with active custom domains, active tier > prototype, or an explicit "production" flag.

2. **Pin is a chicken-and-egg gate — IMPORTANT.** `run402 projects pin <id>` returns 404 "Project not found or not active" for archived projects. Once a project is archived, the only path forward is re-provisioning a new `project_id`. **Workaround:** pin every production project within seconds of `projects provision` (build the pin call into the bootstrap script). **Suggested fix:** allow `pin` to atomically restore-from-archive AND set `pinned=true` in one operation, so owners who discover archived production projects can recover without re-provisioning.

3. **Archive leaves custom-domain rows orphaned and unreleasable by the wallet owner — IMPORTANT.** The archive cascade is inconsistent: it deletes `internal.subdomains` rows but preserves some `internal.domains` rows (custom root-domain registrations). On 2026-04-14, the eagles and silver-pines domain rows were cleaned during archive, but `barrio.kychon.com` and `kychon.com` were left pointing at their archived `project_id`. The archived project's `service_key` is no longer accepted (401), so `run402 domains delete` fails — and no admin tool exists to release domains owned by the wallet but bound to an archived project. **Workaround:** use admin SQL via a different active project (`POST /projects/v1/admin/<active-id>/sql` → `UPDATE internal.domains SET project_id=<new> WHERE domain=<...>`) to surgically rebind the orphan to a fresh project, preserving the Cloudflare custom-hostname ID and avoiding DNS churn. **Suggested fix:** consistent cleanup policy — either drop all `internal.domains` rows on archive, OR expose `run402 domains release-orphan <domain>` that accepts SIWX auth from the wallet owner (without needing the archived project's service_key).

4. **Inconsistent per-project archive cleanup behavior — IMPORTANT.** During the 2026-04-14 archive cascade, two projects' custom-domain rows were released (eagles, silver-pines) and two were preserved (barrio, marketing/kychon.com). We don't know why the behavior diverged — all four were archived under identical conditions. This makes recovery unpredictable: a reprovision+rebind operation may hit either code path and needs to handle both. **Workaround:** query `internal.domains` before provisioning to detect orphans, then choose between `run402 domains add` (if free) or admin SQL UPDATE (if orphaned). **Suggested fix:** deterministic cleanup policy documented in the archive function, tested in integration tests.

5. **✅ FIXED (2026-04): Auth signup lifecycle hook.** Previously (MEDIUM): no server-side post-signup hook; clients had to call a function explicitly from JS after the OAuth callback, which was fragile (tab close races). Run402 now ships a lifecycle-hook system: a deployed function named `on-signup` is automatically invoked after first signup with `{ user: { id, email, created_at } }` payload. Kychon's `functions/on-signup.js` consumes it directly. This is the canonical closed-loop example referenced in saas-factory F27 — product demand surfaced in this file → platform capability shipped → product workaround replaced.

6. **Email templates too rigid — HIGH.** (See `docs/spec.md` Gap 1 for full detail.) 3 fixed templates (`project_invite`, `magic_link`, `notification`); the `notification` template caps at 500 chars plain text. Kychon needs welcome emails, RSVP confirmations, renewal reminders, rich announcements, weekly digests. **Workaround:** edge function calling an external email API. **Suggested fix:** `raw_email` endpoint accepting subject + HTML body with abuse guardrails.

7. **`getUser(req)` missing email — MEDIUM.** (See `docs/spec.md` Gap 6.) Password-auth users' JWT claims don't include email, breaking the on-signup flow without client-provided data (trust issue). **Workaround:** client-side JS passes email to the on-signup function body. **Suggested fix:** include `email` in `getUser()` response consistently across all auth modes.

8. **No full-text search helper — MEDIUM.** (See `docs/spec.md` Gap 3.) PostgREST supports Postgres FTS (`?body=fts.word`) but needs `tsvector` columns + GIN indexes in the tenant schema. **Workaround:** Kychon adds them itself. **Suggested fix:** documented pattern or helper migration for common FTS setups.

9. **No batch REST operations — LOW.** (See `docs/spec.md` Gap 5.) Approving 12 members = 12 PATCH requests; CSV import of 200 members = 200 POSTs or a dedicated edge function. **Workaround:** edge function with `db.sql()` for bulk updates. **Suggested fix:** PostgREST-compatible batch PATCH/POST endpoints.

10. **10MB file upload limit — LOW (Kychon v1).** (See `docs/spec.md` Gap 4.) Fine for photos/PDFs/documents; limits video and large presentations. Not blocking for association/HOA/church beachheads; worth revisiting for photographer/course-portal variants.

11. **SQL pattern filter blocks `SET role` as a column name — LOW.** (See `docs/spec.md` Gap 7.) The injection filter `\bSET\s+(search_path|role)\b` false-positives on `UPDATE members SET role = 'admin'` where `role` is a column, not the Postgres command. **Workaround:** use `db.from('members').update({role:...}).eq(...)` which bypasses the SQL filter. **Suggested fix:** smarter parsing that distinguishes `SET ROLE` (statement) from `SET role =` (column assignment inside UPDATE).

12. **Static file caching has no cache-busting — LOW.** (See `docs/spec.md` Gap 8.) Static files ship with `cache-control: public, max-age=3600` — post-redeploy, browsers serve stale CSS/JS for up to an hour. **Workaround:** manual `?v=N` query strings on asset references (see `public/css` references in Astro build). **Suggested fix:** content-hash appended to file URLs at deploy time, OR ETag-based revalidation with shorter `max-age`.
